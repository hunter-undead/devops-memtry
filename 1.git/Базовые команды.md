# Файлы 
git add . // добавить файлы для пуша
git commit -m "message" // комментарий, чтобы знать, что было добавлено/изменено при пуше
git push // закинуть с локалки в сетевой репо файлы

# Ветки
git branch -a // Просмотр ВСЕХ веток
git checkout (имя_ветки) // Переключение на нужную ветку
git pull origin (имя_ветки) // синхронизируем изменения
git checkout -B (имя-ветки) // создание новой ветки (чтобы пушить в ветку, соответственно = git push origin (имя) )

## Предложение внести свои изменения в основную ветку
pull request // для GitHub
merge request // для GitLab/BitKeeper

Если одобрено, то:
git merge // При слиянии создает доп. merge commit. Видно как ветки объединялись, история разветвления.
git rebase // Переносит наши коммиты на самый свежий ветки main. История линейной и чистой.

Разница в том, что в merge могут хламиться commit. А rebase делает так, словно вы уже начали работать, после свежих обновлений.


# Практика. Как в итоге слить ветви
1. git checkout master — переключаетесь на целевую ветку (туда, куда будете вливать изменения)

2. git pull origin master — подтягиваете свежие изменения из удаленного репозитория, чтобы избежать конфликтов

3. git merge test — вливаете изменения из ветки test в текущую ветку master

4. git push origin master — отправляете результат на сервер
